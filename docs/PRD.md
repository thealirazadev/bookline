# PRD: bookline

## What we're building

bookline is a self-hosted appointment booking app for a single host. The host signs in, defines
event types (name, duration, buffer before/after, minimum notice, maximum days ahead) and weekly
availability with per-date overrides and blackout dates, then shares a public booking page. A
visitor opens that page, sees open slots rendered in their own timezone (auto-detected, manually
overridable), picks one, and books it with their name and email. Both sides receive a confirmation
email with a valid .ics invite; a reminder email goes out before the appointment; the visitor can
cancel or reschedule through signed links in the email; and the host can subscribe any external
calendar to a read-only iCal feed of their bookings. All times are stored in UTC, availability
rules live in the host's IANA timezone, and slot math is correct across DST transitions.

## Target user

- Primary: the host — a freelancer, consultant, or barber who takes appointments and wants a
  booking page they control, without a SaaS subscription. They set up availability once, tweak it
  with overrides, and check a dashboard of upcoming bookings.
- Secondary: the visitor booking a slot. They may be in any timezone, expect to see times in their
  own clock, and expect a calendar invite and working cancel/reschedule links. They never create an
  account.

## Core features (prioritized)

### P0 — Timezone-correct slot engine
The core of the product. Weekly availability rules are stored as minutes-of-day in the host's IANA
timezone; slots are generated by projecting those rules into concrete UTC instants for each date
and rendered in the visitor's timezone. DST transition days are handled explicitly (skipped local
hours produce no slots; repeated local hours produce each wall time exactly once), and availability
windows that cross midnight work. This is a hard requirement of Phase 1, delivered as a pure,
heavily unit-tested library before any UI exists.

### P0 — Concurrency-safe booking flow
A visitor books a slot with name and email. If two visitors submit the same slot simultaneously,
exactly one wins — enforced by a PostgreSQL exclusion constraint on the booking's occupied time
range, not by an application-level check. The loser gets a friendly "that time was just taken"
response with refreshed slots for the day. Hard requirement of Phase 2.

### P0 — Calendar interop (.ics + iCal feed)
Confirmation emails carry a valid RFC 5545 .ics attachment (METHOD:REQUEST). The event UID is
stable across reschedules, SEQUENCE increments on every change, and cancellations send
METHOD:CANCEL for the same UID. A per-host tokenized iCal feed URL lets external calendars
subscribe to all confirmed bookings. Hard requirement of Phase 1 (writer) and Phase 2 (delivery).

### P0 — Public booking page
Month/day slot picker for one event type: a month grid showing which days have open slots, a slot
list for the selected day, timezone auto-detection with a manual override select, and a booking
form (name, email). Days and slots are bucketed by the visitor's timezone.

### P0 — Cancel and reschedule via signed links
Confirmation and reminder emails include a manage link containing an HMAC-signed token bound to
the booking. From the manage page the visitor can cancel (with optional reason) or reschedule;
rescheduling re-runs the full slot engine and the same DB-level double-booking enforcement. No
visitor account or password involved.

### P1 — Host auth and dashboard
Single host account with credentials login (email + password, seeded, no self-signup). Dashboard
lists upcoming and past bookings with invitee details and rendered times in the host's timezone,
and lets the host cancel a booking (which emails the invitee with a METHOD:CANCEL invite).

### P1 — Event type CRUD
Create, edit, deactivate, and delete event types: name, slug, description, duration, buffer
before/after, minimum notice, maximum days ahead, reminder lead time. Deactivated event types
disappear from the public page but keep their history.

### P1 — Availability editor
Weekly rules (multiple windows per weekday, windows may cross midnight), per-date overrides that
replace the weekly rules for that date, and blackout dates that close a date entirely.

### P2 — Reminder emails
A background job sends one reminder email per booking at the event type's configured lead time
before the start, exactly once, only for still-confirmed bookings.

## Non-goals

- No payments, deposits, or paid bookings.
- No teams, round-robin, collective scheduling, or multi-host SaaS — exactly one host account.
- No two-way Google/Outlook OAuth calendar sync; the read-only iCal feed is the v1 interop story.
- No visitor accounts, visitor login, or visitor booking history.
- No SMS or push notifications — email only.
- No recurring appointments or multi-slot bookings.
- No rate limiting or CAPTCHA on the public booking form in v1 (deliberate; revisit if abused).
- No internationalization beyond English; no localized email templates.

## Success criteria per core feature

### Timezone-correct slot engine
- All persisted timestamps are UTC (`timestamptz`); no local time is ever stored for a booking.
- For a host in America/New_York with a rule covering the 2026-03-08 spring-forward night, no slot
  is generated for the skipped 02:00 hour; for the 2026-11-01 fall-back night, each wall time
  yields exactly one slot (earlier offset), never duplicates. Both are named unit tests (see
  `docs/testing.md`).
- A window of 22:00–02:00 generates slots on both sides of midnight, attributed to the weekday the
  window starts.
- Changing the visitor timezone re-renders the same UTC instants under the new clock, including
  moving slots across a date boundary, without refetching generation inputs incorrectly.
- Minimum notice, maximum days ahead, buffers, overrides, and blackouts are all respected in
  generated output, verified by unit tests.

### Concurrency-safe booking flow
- Two parallel POSTs for the same slot result in exactly one 201 and one 409; the database holds
  exactly one confirmed booking (automated integration test against real Postgres).
- The enforcement is a Postgres exclusion constraint on the occupied range; deleting the
  application-level pre-check would still leave double-booking impossible.
- The 409 response includes refreshed slots for that day, and the UI shows a friendly message and
  the updated list.
- Cancelled bookings free their slot (constraint applies only to confirmed rows).

### Calendar interop
- The .ics attachment imports cleanly into Google Calendar, Apple Calendar, and Outlook with
  correct title, UTC times, and organizer/attendee.
- A rescheduled booking keeps its UID, increments SEQUENCE, and updates the event in place in a
  calendar that received the original invite.
- Cancelling sends METHOD:CANCEL with the same UID and a higher SEQUENCE, removing the event.
- The iCal feed URL, subscribed from an external calendar, lists confirmed bookings and drops
  cancelled ones on refresh; the token-less URL returns 404.

### Public booking page
- The month grid marks exactly the days that have at least one open slot in the visitor's timezone.
- Auto-detected timezone matches the browser; changing the override immediately re-renders labels
  and day bucketing.
- A day with no open slots shows an empty state, not an empty white area.
- Slots respect minimum notice ("today" hides slots inside the notice window) and the maximum
  days-ahead horizon.

### Cancel and reschedule via signed links
- The manage link from the email opens the booking without any login.
- A tampered or truncated token gets a friendly invalid-link page, never a stack trace.
- Cancel marks the booking cancelled, frees the slot, and emails both sides with METHOD:CANCEL.
- Reschedule only offers currently valid slots, re-runs the same DB-level enforcement, keeps the
  UID, increments SEQUENCE, and emails both sides the updated invite.
- Manage links for a cancelled or past booking show a clear "no longer available" state.

### Host auth and dashboard
- Correct credentials open the dashboard; wrong credentials show a generic failure message.
- All `/dashboard` routes and host APIs reject unauthenticated requests.
- Upcoming and past bookings are listed in the host's timezone with invitee name, email, event
  type, and status; host cancel triggers the same cancellation flow as the visitor link.

### Event type CRUD
- Created event types appear on the public page immediately; deactivated ones disappear but their
  bookings remain in the dashboard.
- Server-side validation rejects nonsense (duration <= 0, negative buffers, min notice < 0, max
  days ahead < 1, duplicate slug) with field-level messages.

### Availability editor
- Saved weekly rules change generated slots on the public page on next load.
- An override replaces (not merges with) the weekly rules for its date; a blackout date produces
  zero slots regardless of rules; overlapping windows within a day are rejected server-side.

### Reminder emails
- A booking whose reminder time has passed gets exactly one reminder (verified by a claim-then-send
  update, tested for idempotency under a double-fire of the job).
- Cancelled bookings never receive reminders; reminders render times in the invitee's timezone.
